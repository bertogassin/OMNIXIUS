## 4. Где мы не на «лучших» языках

### Главное расхождение: веб-фронт

- **По твоему стеку:** веб = **TypeScript** (типизация, масштаб).
- **В проекте:**
  - **То, чем реально пользуются** — это **app/**:
    - десятки **HTML**-страниц,
    - логика в **обычном JavaScript** (`app/js/api.js`, `professions.js`, `i18n.js`, `utils.js`, `config.js`),
    - без TypeScript, без React/Vue в app/.
  - **TypeScript** есть только в **web/** (React + Vite) — это по сути отдельный, не основной фронт.

Итого: **основной фронт написан не на том языке, что в твоём списке** — он на **JavaScript + HTML**, а не на **TypeScript**.

### Второе: критичный backend

- **По стеку:** в начале **Rust**, потом Go.
- **В проекте:** почти весь backend — **Go** (26 файлов). Rust — один сервис (ранжирование).
- То есть **объём кода** не по порядку «Rust первый»: основной backend на Go, Rust добавлен точечно.

### Третье: мобилка и остальное

- **Swift / Kotlin** — в репо есть, но по 1–2 файла, это заготовки, а не полноценные приложения.
- **Solidity, C++** — в репо вообще нет.

---

## 5. Краткий вывод

- **Совпадает с твоим стеком:** Go, SQL, Python, Spark (Scala), наличие Rust/Swift/Kotlin (хоть и мало), частично TypeScript (только в `web/`).
- **Не совпадает:**
  1. **Основной веб** — на **JavaScript и HTML**, а не на **TypeScript** (и не на TS+React/Vue как главный фронт).
  2. **Основной backend** — по объёму это **Go**, а не Rust; Rust только один сервис.
  3. **Нет в репо:** Solidity, C++.

Если хочешь, чтобы проект реально был «на лучших языках», логичные шаги:
- перевести **app/** (веб-клиент) на **TypeScript** и вынести логику из HTML в TS (или в web/ на React),
- по мере роста выносить критичные части backend в **Rust**, оставляя Go под API и оркестрацию.
